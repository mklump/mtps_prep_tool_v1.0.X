Introduction to BluePhoenix IT Discovery

BluePhoenix uses IT Discovery in its assessment/discovery projects. IT Discovery is an application portfolio management (APM) system that brings together scattered information about the mainframe environment and organizes it into a centralized Metadata Repository. This centralized repository contains up-to-date information about the mainframe environment—existing application software components, their interrelationships, usage, relative cost to operate, and other pertinent facts about development and operations—and provides a corporate view of assets across all IT systems and operational facilities.

IT Discovery forms the first part of a three-phased modernization process named Enterprise IT Modernization (EIM) developed by BluePhoenix. The EIM process can range from providing a combination of cost-effective migration alternatives to a complete replacement of legacy applications—migrating systems to new IT environments, transforming them to implement new requirements, and extending them to new technologies, while reusing proven business logic and preserving the knowledge of the current IT staff. The three phases of EIM are:

  • Phase I: Understanding. This phase focuses on obtaining a comprehensive understanding of the interoperation, interrelationships, and interdependencies of the mainframe environment. The understanding obtained in this phase enables an enterprise to choose the optimal modernization strategy for its mainframe environment.

  • Phase II: Implementation. This phase consists of implementing the modernization strategy that was selected in Phase I. This modernization strategy can include any combination of the following:

    • Migration. This option consists of transferring software and data from a legacy environment—for example, databases, platforms, and languages—to a newer, more modern environment.

    • Remediation. IT remediations are global changes and modifications in software structure, function, or outputs that are required by regulatory changes and new business requirements—for example, moving to a new currency system or numbering convention.

    • Redevelopment. This option consists of reusing the business logic in legacy applications to restructure or reengineer parts of these applications.

  • Phase III: Maintenance. This is the process by which the Metadata Repository is continuously updated to ensure that the knowledge and understanding of the mainframe environment remains up-to-date.

IT Discovery as an Ongoing Process

BluePhoenix approaches assessment/discovery as an ongoing process rather than as a one-time project. Decisions within IT never occur all at the same time. For this reason, it is useful for enterprises to continue to collect IT metadata on an ongoing basis so that their decisions can draw on the latest information about their mainframe environments.

This approach to assessment/discovery enables enterprises to regularly accomplish the following:

  • “Clean up” the mainframe environment. Frequent changes to applications create programs and files that applications never use. Regular use of IT Discovery helps to identify these programs and files.

  • Provide information for each ongoing IT decision. An organization’s application portfolio changes over time. It is important to keep this information up-to-date in order to provide an accurate basis for decision making.

  • Combine static with operational information. Static information can show the overall structure of an organization’s application portfolio. Combining this with operational information—such as CPU utilization—provides an accurate estimate of the cost of systems over time.

  • Carry out impact analysis. IT Discovery identifies and estimates the impact that a change to an application will have on other systems, including files, programs, and JCL (Job Control Language).

  • Refresh the database after carrying out a migration or redevelopment activity. The information contained in IT Discovery forms a crucial starting Metadata Repository for any migration or redevelopment activity. However, once a migration or redevelopment activity is completed, this information is replaced with the post-migration information about the new environment. Thus, an up-to-date IT Discovery is a useful tool for managing the IT environment.

  • Serve as a single source of information for tactical and strategic decision making. IT Discovery enables decision makers to assess the information about all current IT systems as they address various issues relating to the future of their mainframe environments.

  • Serve as a foundation for application modernization. IT Discovery can also serve as the foundation for a number of application modernization activities driven by changes in the external environment—for example, mergers, acquisitions, or department consolidations; application systems integration due to package implementation; the need to support new currencies such as the Euro; and consolidation of servers or data to reduce infrastructure support.

  • Serve as basis for operational cost savings. IT Discovery includes operational data drawn from sources such as system log data, schedulers, and custom-developed monitoring systems. This information enables an enterprise to reduce operational costs in its mainframe environment.

How the IT Discovery Process Works

The IT Discovery process consolidates data located throughout an IT system into the Metadata Repository. The discovery process is divided into three main phases (see Figure 3.1):

  • Collectors. Collectors are processes that analyze various input sources and extract from them the data needed to populate the Metadata Repository.

  • Metadata Repository. The collected data, along with descriptions of this data, is stored in the Metadata Repository.

  • Query and Analysis. This phase of IT Discovery uses a variety of tools to query and analyze the data in the Metadata Repository.

Figure 3.1. IT Discovery framework

IT Discovery Collectors

The collection process is based on a workflow architecture that defines how the collection tools should be used.

Collection Architecture

In collecting either static or dynamic information about an IT environment, it is necessary to access and analyze numerous types of data sources including library systems, source control systems, dictionaries, databases, sequential files, and even reports. In addition, your enterprise will undoubtedly have unique data sources that must be analyzed.

IT Discovery collection provides open and easy-to-modify processes for collecting information across diverse technologies. The IT Discovery process drives the collection process and defines how and when the collection process is to take place. It specifies the necessary workflow steps to ensure that these steps are carried out in the prescribed order whenever a collection process involves multiple collection steps—for example, collecting copybooks prior to collecting main source code.

Collectors

The IT Discovery collectors define the basic entities, attributes, and relationships for fully describing the entities, relationships, and attributes in an organization’s legacy environment. Examples of entities include programs and data sources. Examples of relationships include programs that call other programs, programs and their data sources, and the data sources of data fields. Examples of attributes include database type, data type, and program source location.

IT Discovery’s standard collectors gather information about the following:

  • Language collectors. Programs, maps, copies, and routines. IT Discovery supports various languages including Cobol, PL/I Easytrieve, Natural, ADSO, and COOL:Gen,Mantis.

  • Database collectors. Databases, including VSAM, IMS/DB, DL/I, DB2, Oracle, ADABAS, IDMS, and flat files.

  • Teleprocessing (TP) monitors. Teleprocessing monitors such as CICS and IMS/DC.

  • Job control languages. Job control languages such as OS/390 JCL and REXX.

  • Schedulers. Schedulers, including CA-7, Control-M, and OPCA.

  • Operational information. Operating procedures such as OS/390 SMF, trouble tickets, and change management systems.

A unique cross-reference repository is built from all the above raw data. The Metadata Repository enables the raw data to be converted into strategic information. This strategic information can be used to perform different types of analysis such as:

  • Quantity analysis

  • Complexity analysis

  • Impact analysis

Examples of cross-references include:

  • Which applications, components, and jobs are in use and which are not.

  • How often are different applications and components being used.

  • How many defects are there, in which application, and in which frequency; and which applications or components are problematic.

  • Which components are affected when migrating from a non-relational to a relational database.

  • Which components are affected as a result of implementing new business requirements, new standards, or regulation changes.

  • Which applications/components have large disk and CPU consumption.

  • How files, databases, tables, and records are being used and by which applications.

  • Which applications may require more maintenance than others.

  • Which languages are used by which applications.

  • Which third-party tools are used.

  • Which third-party tools, if any, are not being used although maintenance fees are still being paid to the tool’s vendor.

  • Transactions usage, programs usage, and files usage.

Because these standard collectors are open source products, it is also possible to modify them in order to collect site-specific information. For example, information about the author of a program could be extracted from a comment header at the beginning of each program in accordance with local standards.

Refresh Capability

Collection scripts can be reused to refresh and update the contents of the Metadata Repository in order to reflect ongoing changes. The repository is refreshed on a periodic basis—typically, on a daily basis—rather than in real time because a real-time refresh would not maintain a static view of the repository and therefore would not be able to support multiple reports or queries against the same data.

Although it is possible to rerun the collection process against the total input data in order to update the repository, typically only the “delta” information is extracted. This information is then run through the appropriate collection scripts in order to refresh the content.

Metadata Repository

The Metadata Repository is a relational model that maintains a single, unique, and consistent source of information providing a strategic view of all IT information. Data in this repository can be easily accessed through the IT Discovery Query Facility tool as well as through numerous third-party query and OLAP tools.

Although this IT Discovery Query Facility tool can be used to add information and write collectors for things such as languages that are not commonly used, it can also be used to add non-IT information. For example, a collector could be written that analyzes human resource information in order to collect an organization chart hierarchy so that system costs can be collected and related to objects such as programs and programmers can be analyzed by organizational departments.

IT Discovery Query and Analysis

This phase of IT Discovery consists of querying and analyzing the data in the Metadata Repository.

IT Discovery Query Facility

The IT Discovery Query Facility is a Web-based tool for querying the Metadata Repository (see Figure 3.2). It enables multiple users to quickly and easily create and maintain queries against the repository.

Figure 3.2. IT Discovery Query Facility tool

Following are some of the main features of the IT Discovery Query Facility tool:

  • Repository data model. The IT Discovery Query Facility tool accesses this model in order to enable the user to build his or her query. It uses the information in the model both for accessing the data as well as for formatting and displaying queries. Thus, it enables each installation to customize report headings and to support multi-language reporting.

  • Intranet access. Users can access the IT Discovery Query Facility tool via their company intranet. The tool in turn accesses the Metadata Repository via a standard ODBC connection mechanism. After logging on, the user is presented with a list of repositories and associated categories of queries that he or she is authorized to see and use.

  • User-developed queries. Users can create, test, store, and execute their own queries. These queries can be private to the user who created them or they can be made public and available to all users by an administrator.

  • Output. The user can sift through the output of a query by sorting and filtering it without the additional overhead of executing the query again. The output can be saved in any one of a variety of formats, including XML, Excel®, and HTML. It can also be distributed in a variety of ways—for example, it can be attached to a Microsoft or third-party tool, copied into a document or a presentation, or printed as a report.

Accessing the Metadata Repository with Third-Party Tools

Enterprises can also access the Metadata Repository by using third-party tools that include Online Analytical Processing (OLAP) components. Such third-party tools—for example, SQL Server Analysis Services (SSAS) in Microsoft SQL Server™ 2005 or Cognos PowerCube and Hummingbird—use standard SQL to access the data in the repository.

Figure 3.3. IT Discovery OLAP Query

OLAP queries are particularly useful when both static and dynamic information is being collected. For example, programs and the batch jobs they belong to represent static information, while the CPU usage of these programs represents dynamic information. Both types of information are important for analyzing how costs can be reduced.

Implementing an Assessment Using IT Discovery

As the first step of assessment, the IT-Discovery collectors are installed on the mainframe system. Next, the names of all the input source libraries and the applicable parameters that enable IT Discovery collectors to scan the source code and extract the database information are entered by using ISPF panels.

After all the libraries are named, the IT-Discovery collectors are activated, the libraries are scanned, and the Metadata Repository is generated. The whole process is managed to ensure that all steps are carried out in the correct order.

IT Discovery automatically provides inventory and exception reports. The accuracy of the results depends on the quality of the input. The customer needs to validate the inventory and/or provide the missing components (source, copybooks, procedures, and control cards) and also resolve site-specific details (for example, providing the rules for associating load and source members if the names are not the same).  

Assessment Deliverables

Typical deliverables of the assessment include:

  • Comprehensive set of reports (inventory, exception, cross reference).

  • A Metadata Repository that can be read with standard SQL.

  • Impact analysis. In case the customer would like to analyze a specific application or modernization path. These vary from detailed reports to a complete conversion/modernization strategy and project plan, including an implementation and test plan.

The actual deliverables will depend on the type of project and services requested.
