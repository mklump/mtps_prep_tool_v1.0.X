Introduction to Mainframe Modernization

Infosys views mainframe modernization as an evolutionary path that enterprises need to take to increase the agility of their IT environments and to deliver improved business value. Infosys believes that mainframe modernization can help build a highly flexible IT environment through the adoption of Service-Oriented Architecture (SOA).

Infosys’ assessment/discovery service consists of a rapid portfolio assessment followed by a solution roadmap definition. It enables enterprises to address their concerns about business functionality and costs that are related to their legacy systems.

The Infosys assessment/discovery process is designed as a rapid exercise to quickly build a mainframe modernization strategy. The assessment develops an understanding of the state of an application by measuring its technical health, flexibility, and business value contribution. This understanding suggests the most suitable strategy for modernizing the application. This strategy is then evaluated in terms of the enterprise’s existing business and technology drivers to create a high-level roadmap and business case for legacy modernization.

The assessment/discovery process also contributes to the resolution of problems that are related to integration, collaboration, flexibility, and management of the IT environment. Thus, it enables the enterprise to develop an enterprise-level IT architecture that supports its strategic business objectives.

Infosys divides its mainframe modernization process into two phases:

  • Phase 1: Portfolio Assessment. This phase consists of business and technology assessment. It begins with an inventory of the mainframe environment and the creation of metadata and documentation about applications, data, infrastructure, and operations management within this environment. The applications are then clustered into portfolios. A detailed analysis of each application within a portfolio produces 2 x 2 plots that reveal the state of the application.

  • Phase 2: Solution Roadmap. The plots created during Portfolio Assessment form the basis for generating a hypothesis about the best modernization strategy for the application. This hypothesis is then validated by evaluating the available modernization strategies and taking into consideration their respective technology platforms and dependencies. The validated hypothesis becomes the modernization solution. This phase also performs the relevant Proof-of-Concept (PoC) tests and produces a detailed transformation roadmap to implement the modernization solution.

This two-phase framework for mainframe modernization can be readily customized to accommodate the framework that may already be in use in your IT environment.

Phase 1: Portfolio Assessment

The Portfolio Assessment phase consists of the following three stages:

  • Application repository creation

  • Portfolio segmentation

  • Assessment through indices

Application Repository Creation

This stage consists of creating a repository of information about the applications. The collection of this information is partially automated by using static code analysis tools such as SEEC or Relativity. The repository groups the application data into five categories:

  • Application details. These include the application name, different business units, functionalities, modules, and batch processes.

  • Technology details. These include the platforms, the programming languages (COBOL, PL/1, etc.), and databases (IMS, DB/2, etc.).

  • Code details. These include function points, code size, and exceptions.

  • Support details. These include information about defects, FTE information, and release details.

  • Interface details. These include interaction of the particular application with other applications.

For more information about SEEC, visit their Web site at http://www.seec.com/pages/home.jsp. For more information about Relativity, visit their Web site at http://www.relativity.com/pages/home.asp.

Portfolio Segmentation

The Portfolio Segmentation stage consists of clustering the applications into different portfolios that are based on parameters such as organization structure, business process supported by the applications, the functional layer (channel, transaction, shared services, and education), geography, business unit, products/services, and specific application-level parameters such as dependencies between applications, criticality of application, and technology.

Assessment Through Indices

This stage consists of measuring the performance of the applications against four common problems of the legacy systems: complexity of the integration between applications, huge mainframe license and upgrade costs, inflexibility of applications in adapting to changing business needs, and the poor performance of the applications when measured against critical quality-of-service parameters.

Infosys uses its proprietary method to measure the performance of an application. The method consists of using pre-defined metrics or operands to evaluate an application’s performance. It assigns weighted indices to operands and calculates the values for parameters such as technical health, technical flexibility, functional health, functional flexibility, and business value and cost. These calculated values are translated into several 2 x 2 plots to visualize and hypothesize a modernization approach.

These indices are combined with the business functionality requirements of your enterprise to generate 2 x 2 plots by which Infosys can formulate a hypothesis regarding which modernization strategy would best represent the optimal solution.

For example, if an application is low on technical health and low on business value or criticality, it may need to be re-engineered. If it is low on flexibility, it may need to be re-factored or Web-enabled on the basis of business functionality. If it is high in business value/criticality but low on health, it may need to be replaced with a standard product package or a new system. This approach to assessment ensures that the modernization strategy is aligned with the business objectives and functional requirements of your organization. Phase 2: Solution Roadmap

The Solution Roadmap process consists of the following four stages:

  • Analyze and validate the modernization hypothesis (developed in the Portfolio Assessment phase) through pilots and proof-of-concept tests.

  • Develop the future-state architecture.

  • Develop the business case for the modernization strategy.

  • Create the project plan and implementation roadmap for the modernization strategy.

Validation of Modernization Hypothesis

The modernization hypothesis is analyzed and validated by evaluating each of the four modernization options: re-hosting, re-engineering, integration, and custom development.

The validation process includes the following steps:

  • Identify the applications that can be modernized.

  • Identify the dependencies of these applications—for example, external interfaces.

  • Identify components of applications that require re-engineering before they can be migrated. Such components—for example, assembler programs—may require special strategies in order for them to operate in the new environment.

  • Map the functional and non-functional requirements of the source environment to their respective functionalities in the target environment—for example, security features, batch job scheduler, performance, interfaces, and printing mechanism.

  • Determine the best modernization approach from the available options: re-hosting, re-engineering, integration, and custom development.

  • Perform a pilot modernization (proof of concept):

    • Identify the candidate application.

    • The application should be representative of the various types of complexities in the legacy applications.

    • The application should be relatively small so that the pilot modernization takes a short timeframe to complete.

    • The pilot should validate the hypothesis and resolve most of the technical issues connected with the applications.

The future-state architecture is developed by taking into account the current architecture of the legacy applications and evaluating it with reference to the business objectives and functional requirements of your enterprise.

The final stage in the Solution Roadmap phase is the creation of implementation roadmaps: data migration plan, deployment roadmap, capacity planning, and infrastructure needs.

Tools and Techniques

Infosys uses three main tools in its portfolio assessment:

  • InsITE

  • InFlux

  • InFin-IT

InsITe

InsITe was developed by Infosys as its main tool for portfolio assessment. It has a comprehensive reporting facility that produces application-specific and portfolio-specific reports. These reports include the indices and the 2 x 2 plots that are based on the indices. The reports aid the strategic decision-making process by taking advantage of the comprehensive application repository that it creates in order to assess the indices.

InsITe takes advantage of the code analysis that tools such as Relativity and SEEC provide in order to perform a rapid and in-depth portfolio assessment. InsITe has been successfully used in the modernization projects of various Infosys customers, including a major Australian bank whose portfolio contained more than 2,000 applications.

InFlux

InFlux is a collection of methodologies—techniques, guidelines, and templates—supported by toolsets that establish best practices for developing blueprints for IT solutions. The tools extract and document IT solution requirements and translate them into high-level blueprints for design and development.

InFlux leverages the business rules extraction capabilities of tools such as Relativity and SEEC to map business processes and business rules of the application.

Figure 6.5. Mapping processes-rules using InFlux

InFin-IT

InFin-IT is an Infosys proprietary toolkit that uses a life-cycle view of IT costs to calculate the TCO/ ROI of any IT investment. InFin-IT incorporates all categories of costs into its calculation—for example, system procurement, solution implementation, user education and training, customer training, and maintenance.

Best Practices

The assessment/discovery process must lay down the approach for a multi-year IT transformation around the existing mainframe assets. Based on experience, Infosys has developed certain best practices that are applied when executing the assessment/discovery process:

  • Build a strong business case around cost, risks, and benefits to drive this program.

  • Ownership from senior management of client organization is critical to the success of the exercise.

  • Involve key stakeholders from business and IT throughout the process. 

  • Create an up-to-date view of the mainframe portfolio and take advantage of tools to perform rapid assessments.

  • Balance investments that are required for business improvements with cost savings efforts in order to deliver a self-funding model.

  • Execute pilots to build confidence among internal stakeholders.

  • Set up a dedicated program office jointly between Infosys and the client that focuses on working with internal stakeholders, aligning business and IT imperatives, continuously reviewing the transformation program, building frameworks, and developing standards and guidelines.

Deliverables

The main deliverables in an assessment/discovery project include:

  • Document that addresses scope and business/IT drivers for modernization.

  • Application repository document that contains details about application, technology, code, support, and interface.

  • Mapping and analysis of portfolio of applications that addresses a business process segment for legacy modernization.

  • Recommendations and suggested action plan for each application.

  • Identification of scope of the pilot exercise.

  • High-level requirements for application migration.

  • Completed pilot for migration.

  • Target architecture.

  • Business case analysis on best strategy and roadmap for sequencing application-level migration strategies.

  • Proposal (with estimates) for modernization.

  • Project plan.
